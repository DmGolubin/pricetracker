# Самообновление steering-правил

## Когда обновлять

Steering-файлы в `.kiro/steering/` должны обновляться при следующих событиях:

- Добавлен новый UI-паттерн или компонент, который должен использоваться повсеместно
- Изменена архитектура (новый модуль, новая таблица в БД, новый API-эндпоинт)
- Изменены правила работы (git flow, тестирование, деплой)
- Пользователь явно просит изменить или дополнить правила
- Обнаружено, что правило устарело или противоречит текущему состоянию кода

## Какие файлы обновлять

- `project-architecture.md` — при изменении структуры проекта, добавлении модулей, таблиц, эндпоинтов, компонентов
- `workflow-rules.md` — при изменении процессов работы, UI-паттернов, правил кодирования
- `self-update-rules.md` — при изменении самого процесса обновления правил

## Как обновлять

- Редактировать существующие файлы, НЕ создавать дубликаты.
- Добавлять новую информацию в соответствующий раздел, не дублировать между файлами.
- Убирать устаревшую информацию, а не просто дописывать поверх.
- После обновления steering-файлов — коммитить вместе с основными изменениями.

## Примеры ситуаций для обновления

- Создан новый компонент `volumePicker.js` → добавить в список компонентов в `project-architecture.md`
- Решено использовать TypeScript → обновить правила кодирования в `workflow-rules.md`
- Добавлена новая таблица `notifications` → обновить раздел БД в `project-architecture.md`
- Перешли на другой хостинг → обновить API Base URL и деплой-секцию
- Добавлен новый обязательный UI-паттерн (например, toast-уведомления) → добавить в секцию UI-компонентов в `workflow-rules.md`
